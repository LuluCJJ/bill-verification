# Demo 交付前确认清单

这份清单用于每次给业务或产品试用前做快速自检，重点覆盖当前最新框架：模型抽取票面原始 Key/Value，系统规则完成字段映射和一致性核验，反馈闭环再沉淀到模板配置。

## 1. 环境与启动

- 使用项目内 `.venv/` 管理 Python 依赖，不使用公司电脑的全局 Python 环境。
- 通过 `start_demo.bat` 或等价命令启动 FastAPI 后端和前端静态页面。
- `config.local.json` 只保存在本地，不提交到 Git。
- 模型接口地址、模型名称、API Key 和超时时间可以在“系统设置”页面配置。

## 2. 模型链路

- 文本测试需要返回可读回复，用于确认 endpoint、model 和鉴权可用。
- 图片测试需要使用 `image_url` + base64 data URL，确认公司网关支持多模态输入。
- 正式付款核验需要真实调用模型，而不是只展示预置结果。
- 模型输出应优先包含两层结构：
  - `document_items`：票面原始 Key/Value，尽量保留文档原文，不翻译、不改写、不补全。
  - `extracted_fields`：映射到系统标准字段后的结果，带 `normalized_field`、`raw_label`、`raw_value`、`mapping_source`、`confidence` 和证据。

## 3. 付款核验主流程

- 用户先选择付款任务或上传票面文件。
- 页面并列展示系统支付指令和票面预览。
- 点击“开始 AI 预审”后，系统调用多模态模型抽取票面字段。
- 后端使用模板别名、字段规则和 AI 候选映射组合生成标准字段。
- 风险结果默认突出展示异常项，通过项默认收起或弱化展示。
- “预置结果/期望结果”只用于调试和对照，不能替代真实预审。

## 4. 反馈闭环用例

当前重点演示用例是“入账行别名漏识别”：

1. 选择反馈闭环样例。
2. 点击“开始 AI 预审”，用真实模型抽取票面字段。
3. 如果 `入账行` 没有映射到 `beneficiary_bank`，系统展示缺失风险。
4. 用户在风险卡片上提交优化，给当前模板新增别名 `入账行 -> beneficiary_bank`。
5. 系统保存配置后重新执行真实模型抽取和规则核验。
6. 核验结果中 `beneficiary_bank` 风险应消失或降级。

## 5. 模板与配置

- 模板选择应优先由系统任务带出，AI 可给出建议，人工可以修正。
- 别名配置应有生效范围，避免不同银行、不同国家、不同票据类型之间互相污染。
- 配置不是简单扩写 prompt，而是规则优先、prompt 辅助：
  - 精确别名和固定叫法由后端规则匹配。
  - 位置、版式和注意事项可以写入当前模板 prompt。
  - 模型只给候选映射，最终是否通过由规则核验决定。

## 6. 测试数据

- 合成样例不能使用公司真实票据或真实客户信息。
- 当前可用样例包括中文支票、英文电汇申请、金额差异、账号差异和别名漏识别。
- `docs/word-test-cases/` 下提供中英文 Word 用例，业务可以自行修改字段名称和值，再截图上传测试配置效果。

## 7. 交付前验证

- 运行基础编译检查，确认后端和脚本没有语法错误。
- 重新生成业务汇报 PPT 和 PNG 预览。
- 打开 montage 预览图，确认中文、版式和页数正常。
- 运行前端 JS 语法检查。
- 不提交 `config.local.json`、上传文件、模型返回缓存、反馈运行数据等本地敏感文件。
