# Demo 代码开发前确认清单

## 1. 已确认的工程约定

参考 `contract_handle` 项目，当前 Demo 建议采用项目内 `.venv` 管理 Python 依赖。

约定如下：

- 使用 `.venv/` 作为本项目的 Python 虚拟环境目录。
- `.venv/` 不提交到 Git。
- 后端依赖统一写入 `requirements.txt`。
- Windows 环境下优先通过 `start_demo.bat` 创建虚拟环境、安装依赖并启动服务。
- 启动脚本显式使用 `.venv\Scripts\python.exe -m pip install ...`，避免依赖被装进公司电脑的全局 Python 环境。
- 本地密钥、模型接口地址、样例上传文件、识别结果和反馈结果不提交到 Git。

## 1.1 本轮新确认的产品边界

- 内部模型接口暂按 OpenAI-compatible 设计。
- Demo 需要专门提供模型 API 测试脚本，同时支持纯文本和图片输入测试。
- 样例数据不使用公司真实数据，基于公开常见支票、电汇申请和转账信形态生成合成样例。
- ERP/MAP 系统字段可以比较完整，票面文档只出现字段子集或部分子集。
- 第一版先按整体一致性预审处理，具体字段映射和转换口径后续由 MAP 配置管理。
- Demo 需要从头到尾实现，包括配置管理页面、核验页面、反馈记录，以及“配置规则 + AI 语义映射”的组合设计。

## 1.2 当前 Demo 模式说明

当前页面的“开始 AI 预审”会调用多模态模型提取票面字段，再进入规则核验。预置票面结构化结果仍保留在样例中，主要用于对照、调试和模型不可用时兜底。

模型能力验证通过 `scripts/test_model_api.py` 单独完成：

- 纯文本测试：验证 OpenAI-compatible 地址、模型名和鉴权是否可用。
- 图片测试：验证内部网关是否支持 `image_url` + base64 data URL 的多模态调用。

上传文档预览区已经连接到模型提取接口。当前模型调用要求图片格式为 PNG、JPG 或 WebP；PDF 需要后续增加转图片预处理。

当前页面已提供模型配置与测试区：

- 接口地址。
- 模型名称。
- API Key。
- 超时时间。
- 文本测试。
- 图片测试。
- 调用 AI 提取票面字段。

API Key 只保存到本地 `config.local.json`，该文件不进入 Git。

## 2. 模型接口需要确认

公司内部多模态大模型接口是 Demo 的核心外部依赖，需要优先确认：

- 接口协议：是否 OpenAI-compatible，还是内部 HTTP JSON 接口。
- 模型名称：当前已验证 `Doubao-Seed-2.0-pro`；如切换公司内部其他模型，需要确认实际网关别名。
- 鉴权方式：API Key、Token、Cookie、内网免鉴权或统一网关。
- 图片输入方式：base64、URL、multipart file，还是 message content 中的 image_url。
- 是否支持多图输入。
- 是否支持 PDF 直接输入；如果不支持，Demo 后端需要先转图片。
- 单张图片大小限制。
- 单次请求上下文长度。
- 超时时间、并发限制和限流策略。
- 是否支持 JSON mode、function calling 或结构化输出约束。
- 公司数据合规要求：脱敏票据是否可传入该接口，是否必须内网部署或日志脱敏。

## 3. 样例数据需要确认

Demo 第一阶段最好准备 3-5 组脱敏样例，每组包含：

- 票据图片或 PDF。
- 对应 MAP/AP 系统支付指令 JSON。
- 人工标注的期望字段提取结果。
- 人工标注的一致性判断结果。
- 至少一个通过样例和两个风险样例。

建议样例覆盖：

- 完全一致。
- 金额不一致。
- 账号不一致。
- 币种缺失或不一致。
- 收款方名称近似但可接受。
- 票面出现特殊条款，例如 Not Negotiable 或 A/C Payee Only。

## 4. 字段范围需要确认

MVP 阶段建议先确认是否只做以下字段：

- 付款方名称。
- 付款方账号。
- 付款方银行。
- 收款方名称。
- 收款方账号。
- 收款方银行。
- 金额。
- 币种。
- 付款日期。
- 汇款用途。
- 特殊票面条款。

需要业务进一步确认：

- 中间行、SWIFT、IBAN 是否进入第一版。
- 金额大写是否进入第一版。
- 支票场景中的手写字段是否进入第一版。
- 印章、签字、日期章是否进入第一版。

## 5. 比对规则需要确认

需要在 Demo 前明确以下规则：

- 金额是否必须精确一致到小数位。
- 账号是否去除空格、短横线后精确比较。
- 银行名称是否允许简称、英文名、中文名映射。
- 收款方名称是否允许大小写、标点、公司后缀差异。
- 日期是否允许不同格式但同一天。
- 汇款用途是否做精确一致、包含关系还是语义一致。
- 系统有值但票面没有识别到时，算高风险还是中风险。
- 票面有字段但系统没有字段时，是否只做风险提示。

## 6. 前端交互需要确认

Demo 页面建议先做单任务视图，需要确认：

- 是否需要登录和权限控制；Demo 阶段建议不做。
- 是否需要任务列表；Demo 阶段可先做样例切换。
- 是否需要图片证据框选；第一版可先用页码、区域描述和原文片段，后续再做坐标高亮。
- 是否展示模型原始 JSON。
- 人工反馈按钮有哪些：确认一致、确认不一致、AI 识别错误、忽略、提交优化。
- 风险项是否按高、中、低风险分组展示。

## 7. 后端代码边界

建议第一版后端拆成以下模块：

- `model_client.py`：封装内部多模态模型接口。
- `extractor.py`：组织 prompt，调用模型，解析票面字段。
- `schemas.py`：定义请求、响应和字段 Schema。
- `comparator.py`：金额、币种、账号、名称、日期等规则比对。
- `storage.py`：保存样例、识别结果和人工反馈。
- `main.py`：FastAPI 路由入口。

代码原则：

- 模型负责提取和归一，规则代码负责最终一致性判断。
- 所有模型输出必须经过 JSON Schema 或 Pydantic 校验。
- 模型调用失败时前端要能看到可理解的错误，而不是空白。
- Prompt、字段 Schema、字段别名和比对规则尽量配置化。

## 8. 推荐的第一轮开发顺序

1. 建立后端 FastAPI 骨架和健康检查接口。
2. 建立 `config/field_schema.json` 和模拟支付指令样例。
3. 建立静态 AI 结果样例，不接模型先跑通比对和展示。
4. 实现金额、币种、账号、名称、日期的基础比对。
5. 建立前端单任务 Demo 页面。
6. 接入内部多模态模型接口。
7. 增加人工反馈保存。

## 9. 复杂度评估

建议把 Demo 分成两层理解。

可以一口气做完的轻量闭环：

- 合成样例数据。
- 系统支付指令 JSON。
- 静态 AI 识别结果。
- 字段 Schema、别名和映射规则配置。
- FastAPI 后端接口。
- 单页前端展示。
- 一致性比对结果。
- 人工反馈保存。
- OpenAI-compatible 文本/图片接口测试脚本。

不建议第一轮做成生产级的部分：

- 真正的坐标级字段框选和高亮。
- 多模板自动识别和模板训练。
- 复杂业务口径配置审批流。
- 真实 OCR 与多模态模型结果融合。
- 大规模样本标注和评估报表。
- 与 MAP 的真实接口联调。

因此第一版建议做“完整链路的轻量实现”，让领导和业务能看到从票据、系统数据、AI 识别、配置、比对到反馈的全流程，而不是只看静态页面。
