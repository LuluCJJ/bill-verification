# 当前 Demo 实现说明

## 1. 当前版本定位

当前 Demo 用于验证“基于模板字段配置的权签票据一致性 AI 预审”轻量闭环：

```text
付款任务
→ 系统支付指令
→ 票据文件
→ 已发布模板字段配置
→ 多模态模型按配置字段定向提取
→ 交易系统/MAP 函数包规则核验预览
→ 风险展示
→ 人工反馈
→ 模板字段配置调优并重新真实提取
```

Demo 不使用公司真实票据或真实 MAP 数据，样例全部为合成数据。模型接口按 OpenAI-compatible 协议接入，已在火山方舟 `Doubao-Seed-2.0-pro` 和公司内 OpenAI-compatible 网关上验证文本与图片链路。

当前最新实现将 AI 调用边界前移到模板配置：

- 模板未配置、未启用或未发布时，正式 AI 预审不启动。
- 模板配置包含本次要提取的字段集合、系统来源字段、业务含义、票面别名、位置提示和提取要求。
- 模型只围绕该字段集合做定向提取，避免开放式识别所有票面字段。

模型输出仍保留两层：

- `document_items`：票面原始 Key/Value，尽量逐字保留文档原文。
- `extracted_fields`：当前模板字段集合内的结构化提取结果，用于后续规则核验。

这样可以降低模型改写、补全或翻译票面文字造成的幻觉风险。

## 2. 页面结构

当前前端采用四个标签页组织用户旅程。

### 2.1 付款核验

这是主展示页，面向权签人。

用户流程：

1. 选择一笔样例付款任务。
2. 页面展示系统支付指令。
3. 页面展示票面文档。
4. 页面展示当前模板。
5. 点击“开始 AI 预审”。
6. 后端读取当前模板已发布字段配置。
7. 后端把字段来源、含义、别名、位置提示和提取要求组装进 prompt。
8. 后端调用多模态模型按配置字段定向提取。
9. 后端过滤未配置字段，并保留票面原始 Key/Value 和证据。
10. Demo 后端执行一致性比对预览；生产方案中日期、金额大小写、账号归一等比对规则由交易系统/MAP 函数包负责。
9. 页面默认只展示风险项，通过项可展开查看。
10. 权签人可提交确认、忽略、AI 识别错误或优化反馈。
11. 对“别名漏识别”类反馈，系统可更新别名配置并再次调用真实模型重新提取和核验。

### 2.2 模板字段配置与沙箱调优

面向运营、AI 产品或财务配置人员。

这里按模板维护 AI 是否启动以及定向提取字段：

- AI 启用状态。
- 发布状态：草稿、沙箱测试、已发布。
- 本模板要提取哪些字段。
- 字段来源于哪一个系统字段。
- 字段业务含义。
- 票面常见叫法。
- 位置/模板提示。
- AI 识别要求。

这些内容会进入模型提取 prompt。当前推荐原则是“业务半结构化配置，系统统一组装 prompt”：业务不直接写完整大段 prompt，而是维护含义、别名、位置和要求，系统负责转换为模型可理解的结构化提示。配置应先在沙箱验证，验证通过后再发布到正式核验。

### 2.3 反馈样本

用于承接权签人的人工反馈。

当前版本支持保存反馈到本地 `data/feedback/`，反馈页可展示历史反馈记录。付款核验页内已有“入账行别名漏识别”闭环用例，用于演示权签人如何从风险卡片提交优化并触发后续配置更新。

### 2.4 系统设置

用于一次性配置模型连接信息：

- OpenAI-compatible URL。
- 模型名称。
- API Key。
- 超时时间。
- 文本测试。
- 图片测试。

配置保存到本地 `config.local.json`，该文件被 `.gitignore` 忽略，不会提交到 Git。

## 3. 核心实现

### 3.1 后端

后端使用 FastAPI，主要模块：

- `main.py`：API 路由和静态页面托管。
- `model_client.py`：OpenAI-compatible 模型调用封装。
- `extractor.py`：读取模板字段配置、构造定向提取 prompt、调用模型、解析模型结构化结果；支持 JSON 输出和自然语言列表兜底解析。
- `comparator.py`：Demo 本地比对预览；生产中该能力应由交易系统/MAP 函数包实现。
- `schemas.py`：请求、响应和提取结果 Schema。
- `storage.py`：配置、样例、反馈和本地模型设置读写。

### 3.2 前端

当前前端是轻量原生 HTML/CSS/JavaScript，由 FastAPI 直接托管。

这样做是为了快速验证产品旅程，减少 React/Vite 工程复杂度。后续如果要演进为正式前端，可以把现有四个标签页迁移为 React 路由或组件。

### 3.3 配置文件

- `config/field_schema.json`：标准字段、风险等级、比对方式。
- `config/template_ai_fields.json`：模板级 AI 字段配置，决定模板是否启用 AI、发布状态、提取字段、字段含义、别名和位置提示。
- `config/field_aliases.json`：旧版全局字段常见叫法，当前保留用于兜底和兼容。
- `config/mapping_rules.json`：旧版模板规则和比对归属说明，当前主链路已迁移到 `template_ai_fields.json`。
- `config.local.json`：本地模型连接配置，不提交。

## 4. 样例数据

当前包含四组样例：

1. 国内支票整体一致。
2. 跨境电汇金额不一致。
3. 跨境电汇收款账号不一致。
4. 反馈闭环：票面写“入账行”，初始别名未配置时收款方银行漏映射，提交优化后加入别名并再次真实提取。

样例包括：

- 合成票据 PNG。
- 模拟 MAP/AP 系统支付指令 JSON。
- 预置期望提取结果 JSON。
- 可编辑 Word 测试用例，便于后续改字段、截图生成新样例。

当前主流程会调用真实模型进行票面字段提取；预置结果只用于初始展示、对照和调试，不再替代“开始 AI 预审”的真实模型调用。

## 6. 当前限制

- 反馈样本池当前展示原始反馈记录，尚未做高频问题聚合、规则推荐和审批流。
- 模板识别当前主要依赖样例任务中的 `template_id`，还没有实现“系统自动匹配 + AI 辅助识别 + 人工修正”的完整流程。
- 模板调优规则当前同时影响 prompt 和后端别名兜底映射；后续需要增加精确匹配、包含匹配、模糊匹配等配置边界。
- 文件上传当前主要支持图片预览和模型识别；PDF 转图片能力尚未接入主流程。
- 目前没有用户权限、任务列表、审计记录和 MAP 真实接口。

## 7. 下一步建议

建议下一轮优先做：

1. 模板识别流程：系统带出模板、AI 建议模板、人工修正模板。
2. 反馈样本池：聚合高频错误、转成模板调优建议，并支持配置审批。
3. AI 调优闭环：将反馈一键生成字段别名、位置提示或识别要求，但要明确匹配方式边界。
4. PDF 上传预处理：PDF 转 PNG 后进入模型识别。
5. 更接近 MAP 的系统指令录入和字段分组展示。
