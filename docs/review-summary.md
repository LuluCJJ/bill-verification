# 权签票据一致性 AI 预审方案评审摘要

## 1. 一句话定位

建设一个面向权签环节的票据一致性 AI 预审能力，对纸面支票、转账信、汇款申请书与 MAP/AP 系统支付指令进行自动初筛，提前暴露金额、币种、账号、收款方、银行等关键不一致风险，最终审查责任仍由权签人承担。

## 2. 核心判断

这个需求不应被定义为单纯 OCR，也不应被定义为自动审批，而应定义为“AI 风险预审 + 人工权签确认 + 反馈优化闭环”。

关键难点不在于把票面文字识别出来，而在于：

- 不同国家、不同银行、不同模板上的字段表达差异很大。
- 票面字段需要映射到 MAP/AP 的稳定标准字段。
- 一些字段没有明确 key，只能通过位置、上下文、系统候选值和模板经验判断。
- 后续模板和字段很多，必须支持配置和反馈，不能每次都靠产品发版。

## 3. 产品架构

整体链路建议拆为六层：

1. 票面采集：扫描件、图片、PDF、支票、转账信、汇款申请书。
2. 票面理解：OCR、版面分析、多语言识别、文档分类。
3. 原文抽取：输出票面原始 Key/Value，不改写、不补全。
4. 字段映射：通过别名、模板规则和必要的 AI 语义，将票面字段映射到 MAP/AP 标准字段。
5. 一致性比对：对系统值和票面值进行规则化、语义化比对。
6. 风险展示与反馈：向权签人展示差异、证据、置信度，并沉淀人工反馈。

## 4. 指标策略

不建议只用一个“准确率 80%-90%”来衡量。

建议按风险分层：

- 高风险字段：金额、币种、账号、收款人、付款方、银行，优先保证召回率。
- 中风险字段：日期、用途、地址、银行名称近似，兼顾召回和精确。
- 低风险提示：特殊条款、格式异常、置信度低，优先控制误报。

可对外表达为：高风险字段重召回，低风险提示重精确率。

## 5. 产品分工

MAP/AP 产品负责：

- 提供系统支付指令标准字段。
- 定义权签流程和结果展示。
- 定义哪些字段必须比对、哪些字段允许模糊匹配。
- 保存权签人最终判断和审查记录。

AI 产品负责：

- 票面 OCR、版面理解和字段提取。
- 票面原始 Key/Value 提取和到标准字段的候选映射。
- 输出字段值、置信度、候选值和票面证据位置。
- 提供字段别名、模板规则和反馈样本管理能力。

双方共管：

- 核验字段 Schema。
- 字段风险等级。
- 比对规则标准。
- 反馈闭环机制。

## 6. MVP 建议

第一阶段不要追求覆盖全球全部 200+ 银行模板，建议优先验证闭环。

试点范围：

- 5-10 个高频国家或地区。
- 20-30 个高频银行模板。
- 优先覆盖实际业务量，而不是模板数量。

优先字段：

- 付款方名称、账号、银行。
- 收款方名称、账号、银行。
- 金额、币种、日期、用途。
- 不可转让、只入收款人账户等特殊票面风险字段。

第一版页面需要做到：

- 展示系统值、票面值、比对结果。
- 展示风险等级和差异说明。
- 展示票面证据位置。
- 支持人工确认和反馈。

## 7. Demo 建议

当前 Demo 已实现：

- 后端：Python FastAPI。
- 前端：原生 HTML/CSS/JavaScript，由 FastAPI 托管。
- 存储：本地 JSON 文件。
- 模型：OpenAI-compatible 多模态接口。
- 页面：付款核验、模板调优、反馈样本、系统设置。

Demo 技术路线：

1. 选择样例付款任务。
2. 展示系统支付指令和票面文档。
3. 点击“开始 AI 预审”。
4. 调用多模态模型提取票面字段。
5. 用规则代码完成金额、币种、账号、名称、日期等一致性比对。
6. 默认展示风险项，通过项可展开查看。
7. 支持人工反馈和模板调优。

重要原则更新为：模型优先负责票面视觉理解和原文 Key/Value 抽取，字段映射以配置规则为主、提示词辅助，最终一致性比对由规则代码完成，提升稳定性和可解释性。

## 8. 下一步

建议下一步推进三件事：

1. 确认内部多模态模型接口能力，包括图片输入、JSON 输出、上下文长度、并发、数据合规要求。
2. 准备 3-5 份脱敏票据样例和对应 MAP/AP 系统 JSON。
3. 继续扩展可编辑 Word 测试用例，让业务通过改字段、截图来验证配置能力。
4. 在反馈闭环中明确别名匹配边界，例如精确匹配、包含匹配、模糊匹配，避免配置带来不可控泛化。
